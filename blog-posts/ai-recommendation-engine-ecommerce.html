How to Build an AI Recommendation Engine for E-Commerce

Mar 6, 202612 min read

Amazon attributes 35% of its revenue to product recommendations. Netflix estimates its recommendation engine saves $1 billion per year in customer retention. You don't need their scale to benefit. Even a mid-size e-commerce store with 1,000+ products and 5,000+ monthly visitors can see a 15-30% lift in revenue per user with a well-built recommendation system.

Three Approaches to Recommendations

1. Collaborative Filtering

"Users who bought X also bought Y." This is the classic approach. It works by finding patterns in user behavior — purchases, views, ratings — and recommending items that similar users engaged with. No product knowledge required; it learns entirely from behavioral data.

Strengths: Discovers unexpected connections. A user buying a camera might get recommended a specific tripod that other photographers bought, even if the products share no metadata. Weakness: The cold start problem — new products with no interaction data don't get recommended, and new users with no history don't receive personalized results.

2. Content-Based Filtering

"Because you liked this blue cotton shirt, here are other blue cotton shirts." This approach uses product attributes (category, color, brand, price range, description) to recommend similar items. It works well for new products because it relies on metadata, not behavioral data.

Strengths: No cold start for products. Works immediately for new inventory. Weakness: Creates a filter bubble. It only recommends things similar to what you've already seen, missing opportunities for cross-category discovery.

3. Hybrid (What You Should Build)

The winning approach combines both. Use collaborative filtering as the primary signal, fall back to content-based for cold-start scenarios, and layer in contextual factors (time of day, device, location, season) for real-time personalization.

Technical Architecture

Here's the architecture we implement for e-commerce clients:

  1. Event collection layer. Track user events — page views, add-to-cart, purchases, search queries, wishlist additions — and stream them to a data warehouse. Use Segment, Snowplow, or a custom event pipeline.
  2. Feature store. Pre-compute user features (purchase history, browsing patterns, price sensitivity) and product features (embedding vectors from descriptions, category hierarchies, popularity scores). Store in Redis or a feature store like Feast for low-latency serving.
  3. Model layer. For collaborative filtering, matrix factorization (ALS) or neural collaborative filtering. For content-based, product embeddings using sentence transformers on product descriptions. The hybrid model combines scores from both with learned weights.
  4. Serving layer. A REST API that accepts a user ID and context, retrieves pre-computed candidate lists, applies real-time re-ranking based on current session behavior, and returns the top N recommendations in under 100ms.
  5. Feedback loop. Track which recommendations get clicked and purchased. Use this data to retrain models weekly (batch) and adjust rankings in real-time (online learning).

Solving the Cold Start Problem

Cold start is the biggest practical challenge. Here's how to handle it:

  • New users: Start with popularity-based recommendations (bestsellers, trending). After 3-5 interactions, switch to personalized. Use session-based recommendations (what they've viewed in the current session) as early signals.
  • New products: Use content-based similarity to existing products. If a new jacket shares attributes with a popular jacket, recommend it to the same audience. Boost new product visibility with an exploration factor.
  • Sparse categories: For long-tail products with few interactions, use LLM-generated embeddings of product descriptions to find semantic similarity, even without behavioral data.

Where to Place Recommendations

Placement matters as much as algorithm quality:

  • Product page: "Frequently bought together" (collaborative) and "Similar items" (content-based). This is your highest-converting placement.
  • Cart page: Cross-sell complementary products. "Complete the look" for fashion, "You might also need" for electronics.
  • Homepage: Personalized "Recommended for you" section. For returning users, this should be different every visit.
  • Email: Post-purchase recommendations ("Based on your recent order") and re-engagement ("New items you might like"). These drive 15-25% of recommendation-attributed revenue.
  • Search results: Re-rank search results to prioritize items the specific user is more likely to purchase, based on their history.

Measuring Success

The metrics that matter:

  • Click-through rate (CTR) on recommendation widgets: target 5-15%
  • Conversion rate of recommended products: target 2-5% (vs. 1-2% for non-recommended)
  • Revenue per user lift: target 15-30% increase
  • Average order value increase from cross-sell: target 10-20%
  • Catalog coverage: What percentage of your products get recommended? Low coverage means the system has a popularity bias.

Always A/B test. Run your recommendation engine against a control (random or popularity-based) to measure true incremental impact. The numbers above are achievable but vary significantly by industry and catalog size.

Build vs. Buy

For stores with under 10,000 products and limited engineering resources, managed solutions (Algolia Recommend, AWS Personalize, Dynamic Yield) offer a fast path to good-enough recommendations. For larger catalogs or when recommendations are a core competitive advantage, a custom system gives you full control over the algorithm, data, and user experience.
